# Rishon3D — V1.1 "Scale + City Identity" Pass (design)

Date: 2026-06-18
Branch: `worktree-3d-spike` (isolated; do NOT merge/deploy)
Mode: autonomous-builder. Follows the committed World Detail Pass (`02d49c8`).

## Problem (from user review)

The world detail pass landed: facades, tiled streets, atmosphere, street life. The
core vibe is right, but the build still "reads like a prototype city grid rather than a
finished Roblox-style browser game." The gaps are not "better graphics" — they are
**scale control, readable gameplay surfaces, destinations, and city identity**. The
user filed a 10-point brief; the explicit ordering principle is:

> "We need a global visual scale pass before adding more content."
> "Do this as a V1.1 Scale + City Identity Pass before adding interiors."

## Hard constraints (from the brief)

- Do NOT rewrite the engine. Keep gameplay, physics, controls, map architecture, seeded
  generation, material cache, and instancing.
- Keep it browser-friendly; prefer instanced/merged geometry for repeated props.
- Keep deterministic placement via the existing seeded approach.
- Maintain the Roblox/voxel/blocky style. No photorealism, no heavy textures, no complex
  imported assets.
- Verify with real Playwright screenshots **at 1600x900**.
- `npm test` + `npm run build` + `npm run test:smoke` stay green; no console errors; no
  major FPS regression.

## Strategy: verified waves (not one big burst)

Because correct scale must precede new content, V1.1 is executed as sequenced, each-
verified waves, committing per wave. Later waves build on the verified output of earlier
ones. The wave boundaries map onto the brief's sections:

- **Wave 1 — Global Scale Pass + Camera (Sections 1, 8).** THIS WAVE. UI (minimap,
  phone, app icons, HUD prompt, bottom hint, start/pause menu) shrunk to game-UI scale;
  foreground props (streetlights) shrunk; third-person camera pulled back / framing tuned
  so objects stop filling the frame. Foundational — everything else is judged against it.
- **Wave 2 — Street Identity (Section 2).** Stop lines, lane arrows, traffic lights, sign
  posts, bus stops / taxi stand, parking bays, medians/grass islands, curb ramps;
  deterministic per-road variation. (Crosswalks + double-yellow already shipped.)
- **Wave 3 — Building Facade Purpose (Section 3).** Deterministic facade *types*
  (restaurant / shop / apartment / office / airport terminal) with doors, signs, awnings,
  parapets, entry pads — built on the World-Detail facade system.
- **Wave 4 — Destinations (Section 4).** Airport exterior zone, restaurant street, transit
  station upgrade (supports + platform + stairs for the existing rail).
- **Wave 5 — Phone & Map UX + Instruction flow (Sections 5, partial 6).** Map app with
  player/car/destination markers + selectable waypoint; Places/Jobs apps; tutorial chips +
  small objective label.
- **Wave 6 — Character & NPC life + Vehicle pass (Sections 6, 7).** Faces/hair/accents,
  idle behaviors, outfit variety; car proportions, wheels, lights, color variants.
- **Wave 7 — Content loop (Section 10).** Call car -> drive to airport -> destination
  reached, with the objective system from Wave 5.

This spec details **Wave 1**. Each later wave gets its own plan section + decision log as
it is reached, so we do not over-plan work that earlier waves will reshape.

## Wave 1 — Global Scale Pass + Camera (detail)

Current values (read from source) and Wave-1 targets:

UI (CSS-in-JS in `src/ui/*.ts` + the hint string in `main.ts`):
- **Minimap** `Minimap.ts` `SIZE=180` + inline `width/height:180px` -> ~**138px** desktop
  (responsive: `clamp()` so it is ~116px on narrow/mobile). Keep the canvas backing store
  crisp (render at device size).
- **Phone** `Phone.ts` `.r3d-phone width:300px;height:600px` -> ~**240x480** with
  `max-height:72vh` and a responsive clamp; **app icons** `.r3d-icon 54px` -> ~**42px**;
  scale the car glyph / fonts proportionally.
- **HUD prompt** `Hud.ts` `.r3d-prompt font-size:18px` -> ~**14px**, smaller padding,
  move lower (`top:62%` -> ~`78%`) so it does not cover the player. Speedo 22px -> ~17px.
- **Bottom hint** `Hud.ts` `.r3d-hint` + the long string in `main.ts:40` -> compact
  **control chips** (small pills, key + label) or a short help strip, not one long line.
- **Start/Pause menu** `Menu.ts` title `54px` -> ~**40px**, button `20px` -> ~16px,
  tighter padding so it reads as game UI, not a landing page.

World foreground:
- **Streetlights** `props.ts makeStreetLight`: lantern box `0.5^3` -> ~**0.34^3**, pole
  `CylinderGeometry(0.08,0.1,3.4)` slightly thinner/shorter so foreground lamps stop
  dominating. Keep the point-light budget logic in `World.ts` unchanged.

Camera (`FollowCamera.ts`, `cameraMath.ts`, `Game.ts`, `Engine.ts`):
- **Follow distance**: on-foot `10` -> ~**12.5**, car `11` -> ~**14**; raise `lookHeight`
  / camera a touch so intersections read when driving. Keep `MIN_DIST/MAX_DIST` sane
  (raise `MAX_DIST` if needed for the new defaults).
- **FOV** `Engine.ts` `60` -> keep **58-60** (brief says 55-60); only change if it
  measurably reduces foreground dominance.
- **Clipping**: pull-back reduces building clipping near sidewalks; a full camera-collision
  system is deferred (logged) — Wave 1 just increases distance + tunes defaults.

Out of Wave-1 scope (later waves): roads identity, facade types, destinations, phone
gameplay, NPC/vehicle detail, content loop.

## Verification (Wave 1)

- `npm test`, `npm run build`, `npm run test:smoke` green.
- Playwright at **1600x900**: capture (1) title menu, (2) on-foot city view, (3) phone
  home, (4) pause menu. Confirm UI is visibly smaller and the scene reads less zoomed-in
  vs the pre-wave captures. No console errors.

## Assumptions & Decisions (Wave 1 + program)

- Execute V1.1 as **verified waves, Wave 1 = scale+camera first** — because the user
  explicitly requires a global scale pass before more content; building destinations on an
  un-scaled world would be wasted work.
- Treat the brief's exact px targets as **authoritative defaults** (minimap ~138, phone
  ~240x480, icons ~42, prompt 14px) — chosen midpoints of the user's given ranges.
- Use **responsive `clamp()`** for minimap/phone sizes rather than separate desktop/mobile
  code paths — smallest change that satisfies the "desktop vs mobile" sizes in the brief.
- Bottom hint -> **control chips** (small key+label pills) over a help strip — reads more
  like a game HUD and is the user's first-listed option.
- Camera: **increase follow distance + tune defaults only**; defer a full camera-collision
  / avoidance system to a later wave — because collision is a larger system and the brief's
  immediate complaint (objects filling the frame) is mostly fixed by distance + scale.
- Keep **FOV ~60** unless screenshots show foreground still dominates — avoids changing
  control/feel more than needed (brief: don't change controls/physics unless needed).
- Streetlight is the only foreground world prop called out as oversized; other props
  (benches, trees, hydrants) read acceptably, so Wave 1 shrinks **streetlights only** and
  revisits others if the 1600x900 pass shows them dominating.
- Wire orchestration: file-disjoint subagents — **UI-scale** (`ui/*.ts` + `main.ts` hint),
  **world-prop-scale** (`props.ts`), **camera** (`FollowCamera/cameraMath/Game/Engine`) —
  run in parallel; no shared file, so no edit conflicts in the single worktree.
